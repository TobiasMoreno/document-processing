## Purpose

Receive documents over HTTP multipart, validate their type and size, persist their metadata, and store the original bytes in object storage. Returns a stable identifier that downstream phases (parsing, processing, querying) use to correlate work with the uploaded document.

## Requirements

### Requirement: Upload endpoint accepts multipart document
The system SHALL expose `POST /documents` accepting a single multipart file part named `file`. On success it SHALL respond `201 Created` with a JSON body containing `documentId` (UUID) and `status` set to `UPLOADED`.

#### Scenario: Valid PDF is uploaded
- **WHEN** a client sends `POST /documents` with a `file` part of content-type `application/pdf` whose size is below the configured maximum
- **THEN** the system responds `201 Created` with body `{ "documentId": "<uuid>", "status": "UPLOADED" }`
- **AND** the response is returned without waiting for any downstream processing

#### Scenario: Valid Word document is uploaded
- **WHEN** a client sends `POST /documents` with a `file` part of content-type `application/vnd.openxmlformats-officedocument.wordprocessingml.document`
- **THEN** the system responds `201 Created` with a fresh `documentId` and `status: UPLOADED`

#### Scenario: Valid image is uploaded
- **WHEN** a client sends `POST /documents` with a `file` part of content-type `image/png` or `image/jpeg`
- **THEN** the system responds `201 Created` with a fresh `documentId` and `status: UPLOADED`

### Requirement: Upload validates content type
The system SHALL reject uploads whose effective content-type is not one of: `application/pdf`, `application/msword`, `application/vnd.openxmlformats-officedocument.wordprocessingml.document`, `image/png`, `image/jpeg`. The effective content-type is determined by sniffing the file bytes; if the sniffed type does not match the declared multipart `Content-Type`, the upload SHALL be rejected.

#### Scenario: Unsupported content type
- **WHEN** a client uploads a file whose sniffed content-type is `application/zip`
- **THEN** the system responds `400 Bad Request` with body `{ "error": "INVALID_CONTENT_TYPE", "message": "<safe message>" }`
- **AND** nothing is persisted in the database or in object storage

#### Scenario: Declared content type does not match file bytes
- **WHEN** a client uploads a `.zip` file but declares `Content-Type: application/pdf` in the multipart part
- **THEN** the system responds `400 Bad Request` with `error: INVALID_CONTENT_TYPE`
- **AND** nothing is persisted

### Requirement: Upload enforces size limit
The system SHALL reject uploads whose size exceeds 10 MB and SHALL reject empty uploads.

#### Scenario: File exceeds maximum size
- **WHEN** a client uploads a file larger than 10 MB
- **THEN** the system responds `400 Bad Request` with `error: FILE_TOO_LARGE`
- **AND** nothing is persisted

#### Scenario: Empty file
- **WHEN** a client uploads a file part with zero bytes
- **THEN** the system responds `400 Bad Request` with `error: EMPTY_FILE`

#### Scenario: Missing file part
- **WHEN** a client sends `POST /documents` without a `file` multipart part
- **THEN** the system responds `400 Bad Request`

### Requirement: Document metadata is persisted on successful upload
On a successful upload the system SHALL persist a `Document` row containing `id`, `originalFilename`, `contentType` (effective sniffed type), `sizeBytes`, `storagePath`, `status = UPLOADED`, `createdAt`, `updatedAt`. The `id` SHALL be a UUID generated by the application and SHALL match the `documentId` returned in the response.

#### Scenario: Metadata row is written
- **WHEN** a valid file is uploaded
- **THEN** exactly one row exists in the `document` table with the returned `documentId`
- **AND** its `status` is `UPLOADED`
- **AND** its `original_filename`, `content_type` and `size_bytes` reflect the uploaded file
- **AND** `created_at` and `updated_at` are set to the current timestamp

### Requirement: Upload is atomic across storage and database
The system SHALL ensure that on a successful response there is both a stored object and a corresponding metadata row. If the metadata insert fails after the object was stored, the system SHALL attempt to delete the stored object on a best-effort basis before responding with an error.

#### Scenario: Database failure after object upload
- **WHEN** the file has been uploaded to object storage but the database insert fails
- **THEN** the system attempts to delete the stored object
- **AND** the system responds `500 Internal Server Error` with `error: INTERNAL_ERROR`
- **AND** no `Document` row exists for that upload

#### Scenario: Object storage failure
- **WHEN** the file fails to upload to object storage
- **THEN** the system responds `500 Internal Server Error` with `error: STORAGE_ERROR`
- **AND** no `Document` row is created

### Requirement: Document identifiers are stable and unique
Every accepted upload SHALL receive a distinct UUID. The identifier SHALL be generated before any I/O against object storage so that it can be used as part of the object key.

#### Scenario: Two uploads receive distinct IDs
- **WHEN** two valid uploads are processed
- **THEN** they receive two different `documentId` values

### Requirement: Error responses do not leak internals
Error responses SHALL be JSON objects with `error` (machine-readable code) and `message` (safe human-readable text). The system SHALL NOT include stack traces, file system paths, database error text, or internal hostnames in error responses.

#### Scenario: Internal failure is reported safely
- **WHEN** any internal exception occurs during upload
- **THEN** the response body contains only `error` and `message` fields
- **AND** `message` does not contain stack traces or internal paths
